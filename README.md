# Gemini Developer Use Cases вЂ” Hackathon Starter Kit

Three production-inspired demos that use the **Gemini API** to solve real software
developer pain points. Each use case reads clean structured data, calls Gemini with
a typed JSON schema, validates the response with Pydantic, and saves results for
downstream consumption.

---

## Use Cases

| # | Name | Pain Point | Gemini Capability |
|---|------|-----------|-------------------|
| 1 | **Ticket Triage** | Raw support tickets need priority, category, team routing | Structured JSON output |
| 2 | **Sprint Planner** | Vague requirement text needs stories, criteria, test cases | Nested structured output |
| 3 | **Log Explainer** | Noisy logs need root-cause analysis and runbooks | Evidence-grounded analysis |

## Setup

### Prerequisites

- Python 3.10 or higher
- A Gemini API key вЂ” get one free at [https://aistudio.google.com](https://aistudio.google.com)

### Install dependencies

```bash
pip install -r requirements.txt
```

### Set your API key

```bash
# Linux / macOS
export GEMINI_API_KEY=your_api_key_here

# Windows PowerShell
$env:GEMINI_API_KEY="your_api_key_here"
```

Or copy `.env.example` to `.env`, fill in the key, and source it:

```bash
cp .env.example .env
# edit .env
source .env        # Linux / macOS
```

## How to Run

### Step 0 вЂ” Prepare datasets (run once)

```bash
python scripts/prepare_datasets.py
```

This reads `data/raw/*.csv`, cleans each file, and writes `data/clean/*.csv`
plus a validation report in `docs/dataset_validation_report.json`.

---

### Use Case 1 вЂ” Ticket Triage

```bash
# Quick demo (3 tickets)
python src/use_cases/ticket_triage.py --limit 3

# Larger batch
python src/use_cases/ticket_triage.py --limit 50 --batch-size 10
```

**What you see:**
- Per-ticket: `priority`, `likely_team`, `short_summary`, `suggested_reply`, `confidence`
- Output files: `output/ticket_triage/triage_results.jsonl`

---

### Use Case 2 вЂ” Sprint Planner

```bash
# Quick demo (1 requirement)
```

**What you see:**

## Validation Checklist

Every script performs validation at two layers:

| Layer | What is checked |
|-------|----------------|
| **Input (pre-call)** | Required columns present, minimum row count, non-blank text fields, minimum text length |
| **Output (post-call)** | Pydantic schema (field types, literal values, constraints), business rules (e.g., critical priority needs high confidence) |

Rows that fail input validation are written to `skipped_*.json`.  
Outputs that pass schema validation but trigger business warnings are written to `output_warnings.json`.

## Hackathon Demo Tips

1. **Run `--limit 1` or `--limit 3`** on stage so the demo completes in seconds.
2. **Show "before" (raw messy text) vs "after" (structured JSON)** side by side.
3. **Open a generated Markdown file** вЂ” judges immediately see the artifact value.
4. **Show `output_warnings.json`** вЂ” it demonstrates that you built guardrails, not just a prompt.
5. **Talk about the data pipeline** вЂ” `prepare_datasets.py` shows real engineering discipline.

## Extending the Project

| Idea | How |
|------|-----|
| Add a Streamlit UI | `pip install streamlit`, wrap each script as a Streamlit page |
| Connect to a real Jira | Use the Jira REST API to create stories from `sprint_plans.jsonl` |
| Ingest real PagerDuty logs | Replace the CSV with a PagerDuty webhook payload |
| Add batch retry logic | Wrap the API call in a retry loop with exponential backoff |
| Fine-tune prompts by app | Add a `prompt_templates/` folder with per-product-area templates |
