# Gemini Developer Use Cases — Hackathon Starter Kit

Three production-inspired demos that use the **Gemini API** to solve real software
developer pain points. Each use case reads clean structured data, calls Gemini with
a typed JSON schema, validates the response with Pydantic, and saves results for
downstream consumption.

---

## Use Cases

| # | Name | Pain Point | Gemini Capability |
|---|------|-----------|-------------------|
| 1 | **Ticket Triage** | Raw support tickets need priority, category, team routing | Structured JSON output |
| 2 | **Sprint Planner** | Vague requirement text needs stories, criteria, test cases | Nested structured output |
| 3 | **Log Explainer** | Noisy logs need root-cause analysis and runbooks | Evidence-grounded analysis |

### Data flow for each use case

```
Raw CSV (messy)
    │
    ▼
prepare_datasets.py
    │  normalize_whitespace / normalize_case / safe_fill / deduplicate
    ▼
Clean CSV
    │
    ▼
Application script
    ├── Pre-validate row (require_columns, min length, blank checks)
    ├── Build structured prompt (context + raw text)
    ├── Call Gemini (response_mime_type=application/json + Pydantic schema)
    ├── Post-validate (Pydantic schema + business rules)
    └── Write .jsonl results + Markdown artifacts + warnings log
```

## Setup

### Prerequisites

- Python 3.10 or higher
- A Gemini API key — get one free at [https://aistudio.google.com](https://aistudio.google.com)

### Install dependencies

```bash
pip install -r requirements.txt
```

### Set your API key

```bash
# Linux / macOS
export GEMINI_API_KEY=your_api_key_here

# Windows PowerShell
$env:GEMINI_API_KEY="your_api_key_here"
```

Or copy `.env.example` to `.env`, fill in the key, and source it:

```bash
cp .env.example .env
# edit .env
source .env        # Linux / macOS
```

## How to Run

### Step 0 — Prepare datasets (run once)

```bash
python scripts/prepare_datasets.py
```

This reads `data/raw/*.csv`, cleans each file, and writes `data/clean/*.csv`
plus a validation report in `docs/dataset_validation_report.json`.

---

### Use Case 1 — Ticket Triage

```bash
# Quick demo (3 tickets)
python src/use_cases/ticket_triage.py --limit 3

# Larger batch
python src/use_cases/ticket_triage.py --limit 50 --batch-size 10
```

**What you see:**
- Per-ticket: `priority`, `likely_team`, `short_summary`, `suggested_reply`, `confidence`
- Output files: `output/ticket_triage/triage_results.jsonl`

---

### Use Case 2 — Sprint Planner

```bash
# Quick demo (1 requirement)
python src/use_cases/sprint_planner.py --limit 1

# Larger batch
python src/use_cases/sprint_planner.py --limit 20
```

**What you see:**
- Per-requirement: `epic_name`, `stories[]`, `acceptance_criteria[]`, `test_cases[]`, `estimate_points`
- Output files: `output/sprint_planner/sprint_plans.jsonl` + `output/sprint_planner/markdown/*.md`

---

### Use Case 3 — Log Explainer

```bash
# Quick demo (1 incident)
python src/use_cases/log_explainer.py --limit 1

# Filter by environment
python src/use_cases/log_explainer.py --limit 20 --env prod
```

**What you see:**
- Per-incident: `probable_root_cause`, `confidence`, `evidence_lines[]`, `immediate_actions[]`, `incident_severity`
- Output files: `output/log_explainer/incident_analyses.jsonl` + `output/log_explainer/runbooks/*.md`

## Validation Checklist

Every script performs validation at two layers:

| Layer | What is checked |
|-------|----------------|
| **Input (pre-call)** | Required columns present, minimum row count, non-blank text fields, minimum text length |
| **Output (post-call)** | Pydantic schema (field types, literal values, constraints), business rules (e.g., critical priority needs high confidence) |

Rows that fail input validation are written to `skipped_*.json`.  
Outputs that pass schema validation but trigger business warnings are written to `output_warnings.json`.

## Hackathon Demo Tips

1. **Run `--limit 1` or `--limit 3`** on stage so the demo completes in seconds.
2. **Show "before" (raw messy text) vs "after" (structured JSON)** side by side.
3. **Open a generated Markdown file** — judges immediately see the artifact value.
4. **Show `output_warnings.json`** — it demonstrates that you built guardrails, not just a prompt.
5. **Talk about the data pipeline** — `prepare_datasets.py` shows real engineering discipline.

## Streamlit UI

A multi-page Streamlit app wraps all three use cases with a browser-based interface.

### Run locally

```bash
pip install -r requirements.txt
export GEMINI_API_KEY=your_key_here   # Windows: $env:GEMINI_API_KEY="..."

streamlit run app.py
```

Your browser will open at **http://localhost:8501** automatically.

### Pages

| Page | File | What it does |
|---|---|---|
| 🏠 Home | `app.py` | Overview + architecture diagram |
| 🎫 Ticket Triage | `pages/1_ticket_triage.py` | Paste ticket or load from CSV → priority, team, reply |
| 📋 Sprint Planner | `pages/2_sprint_planner.py` | Paste requirement → stories, criteria, test cases |
| 🔍 Log Explainer | `pages/3_log_explainer.py` | Paste log chunk → root cause, evidence, runbook |

### Key UI features

- **API key in sidebar** — enters once, persists across all pages for the session.
- **Dual input modes** — type/paste live text, or pick any row from the 1,200-row clean CSV.
- **Structured output display** — metrics, color-coded severity badges, expandable story cards.
- **Download buttons** — every result exports as JSON and/or Markdown (Confluence-ready).
- **Input & output validation** — errors shown inline with Streamlit `st.error()`.
